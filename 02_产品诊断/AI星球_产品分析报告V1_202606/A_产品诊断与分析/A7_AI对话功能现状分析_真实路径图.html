易用性评测范围建议（基于真实流程）

P0 - 核心路径（必测）

  • Flow 1: 从首页找到对话入口并成功进入
    ✓ 成功标准：15秒内找到小星卡片并点击 | 观察：视线扫描路径、是否遗漏卡片
  • Flow 2: 完成一次语音输入对话（按住-说话-松开）
    ✓ 成功标准：第一次按住说话成功识别 | 观察：是否理解「按住」交互、松开时机
  • Flow 3: 切换到文本模式并发送消息
    ✓ 成功标准：找到切换图标并成功发送 | 观察：IP缩小是否引起困惑
  • Flow 4: 切换IP并完成一轮对话
    ✓ 成功标准：30秒内找到设置入口并切换 | 观察：是否理解三个IP的差异
  • Flow 8: 危机响应界面理解度测试（截图呈现方式）
    ✓ 成功标准：用户能识别12356为可用求助热线 | 观察：吉塔+小星双来源是否引起困惑；须独立伦理设计，不得在常规测试中真实触发

P1 - 重要功能（建议测）

  • Flow 5: 识别聊愈室气泡入口 → 进入沉浸场景 → 完成形象/声音切换 → 结束对话
    ✓ 成功标准：注意到按钮并主动点击 | 观察：是否发现右上角两个圆形按钮、是否尝试语音指令
  • Flow 6: 响应换装提示并完成一次换装
    ✓ 成功标准：找到解锁的服饰并成功更换 | 观察：是否觉得弹窗打断
  • Flow 7: 理解「以下为新对话」标签含义 → 主动找到清空入口
    ✓ 成功标准：能解释标签意图并完成清空操作 | 观察：跨IP混排头像是否造成困惑
  • 深入话题链接: 点击对话中的蓝色链接（ai对话流程.png左下）
    ✓ 成功标准：识别出可点击链接并查看详情 | 观察：是否注意到链接存在

关键观察点

  • 首页小星卡片的发现性（是否注意到对话入口？）
  • 语音按钮的交互理解度（按住说话是否清晰？）
  • 键盘弹起后IP缩小的感知（是否觉得IP消失了？）
  • IP切换界面的三个角色区分度（能否快速理解差异？）
  • 聊愈室气泡按钮的视觉辨识度（能否与普通对话气泡区分？）
  • 进入聊愈室后的场景沉浸感（客厅背景是否产生情境转换感知？）
  • 形象/声音切换面板的发现性（右上角两个圆形按钮是否易找？）
  • 语音指令"切换形象""切换声音"的可发现性（用户是否会主动尝试语音指令？）
  • 换装解锁提示的打断感（是否觉得被打扰？）
  • 「以下为新对话」标签是否被理解为会话隔断？
  • 跨IP混排头像是否造成「谁在回我话」的认知混乱？
  • 危机响应卡片的情绪安全感（是否觉得被关怀还是被监视？）

测试方法建议

  • 任务完成率：P0路径必须≥90%，P1路径≥70%
  • 任务完成时间：记录每个Flow的操作耗时，识别卡点
  • 错误率：记录误操作次数（点错/找不到按钮/理解错误）
  • 主观评分：任务后询问SUS量表（系统可用性量表）
  • 观察法：记录用户的语言反馈（"这是什么？""怎么操作？"）
  • 眼动追踪（可选）：识别视觉盲区和扫描路径
  • Flow 8专项：须使用截图展示替代真实触发，独立伦理审查
用户体验数据采集埋点方案（按维度）

① 用户操作埋点

  • dialog_entry_tap — 点击首页小星卡片，记录页面停留时长 → 衡量发现耗时
  • voice_button_hold_start / end — 按住/松开语音按钮，记录时长、识别成功与否
  • input_mode_switch — 语音↔文本切换，记录切换频率
  • ip_switch_tap — 触发IP切换，记录从哪个IP切到哪个IP
  • chat_clear_tap — 点击「清空聊天记录」及确认/取消结果
  • link_in_reply_tap — 点击对话回复中的蓝色深入话题链接

② 系统响应埋点

  • voice_recognition_latency — 松开按钮到文字上屏的耗时（毫秒）
  • ip_reply_latency — 发送消息到AI第一个字符出现的耗时
  • keyboard_show — 键盘弹起时间，同步记录IP是否缩放
  • session_tag_display — 「以下为新对话」标签曝光，记录用户是否继续滚动/停顿
  • costume_unlock_modal_show — 换装弹窗触发时刻

③ API/后台埋点

  • voice_asr_success_rate — 语音识别成功率（后台），区分首次/重试
  • ip_reply_error_rate — AI回复失败率（超时/空响应）
  • emotion_score_at_liaoyu_trigger — 聊愈室按钮弹出时的情绪分值（APK emotionScore字段）
  • crisis_keyword_hit — 危机词命中事件，记录词类别（不记录原文），同步触发L3后台通知
  • session_new_tag_insert — 新会话标签插入时机（后台判断依据）

④ 决策分支 & 特殊功能埋点

  • liaoyu_entry_tap / skip — 聊愈室气泡按钮点击或被忽略（停留超5秒未点=skip）
  • liaoyu_avatar_switch / voice_switch — 聊愈室内形象/声音切换操作
  • crisis_card_hotline_tap — 12356热线被点击（极高优先级指标）
  • crisis_card_dismissed — 安全卡片被划走/忽略（不显示具体内容，只记录行为）
  • cross_ip_history_scroll — 切换IP后用户向上滚动翻看历史记录的深度

⑤ 流程结束 & 留存埋点

  • dialog_session_duration — 单次对话会话时长（入口到退出）
  • dialog_round_count — 单次会话对话轮数
  • liaoyu_end_tap — 点击「结束对话」退出聊愈室，记录在聊愈室内停留时长
  • next_day_return_rate — 完成AI对话后次日回访率（模块留存核心指标）
  • dialog_exit_method — 退出方式（主动返回首页 / 切Tab / App后台）

⑥ 易用性专项埋点

  • first_voice_attempt_success — 首次语音输入是否一次成功（P0 Flow 2核心指标）
  • entry_find_time — 首页到首次点击小星卡片的间隔秒数
  • ip_selection_browse_time — IP选择界面的浏览时长（衡量差异理解）
  • session_tag_pause_duration — 用户在「以下为新对话」标签处的停顿时长（衡量理解度）
  • cross_ip_confusion_signal — 切换IP后用户在1分钟内再次打开设置菜单（推测为困惑信号）
埋点实施优先级：Flow 8危机词命中（crisis_keyword_hit）+ 12356热线点击（crisis_card_hotline_tap）须优先上线，属于安全合规监控指标。Flow 2语音首次成功率（first_voice_attempt_success）为P0易用性快赢指标。其余按Phase 1→2分批上线，对应升级路线图中的反馈埋点①–⑨节点。
易用性测试实施方案（分两类采集方式）

一、用研型采集（一次性测试活动）

▸ 怎么实施

找 5-8 个真实学生（建议初中/高中各一组），在学校里做1对1有主持人的任务测试。不要说"你来试用这个App"——要给具体场景：「假设你最近压力很大，打开App，想跟一个AI聊聊」。让他们边操作边出声说想法（出声思考法）。

▸ 在哪里采集

学校空教室或会议室，用他们平时用的设备（同款平板）。不要模拟实验室环境，越接近真实使用场景越好。

▸ 以什么形式采集

  • 录屏 + 前置摄像头同步录制（表情反应）
  • 研究员在旁记录卡点和口头反应
  • 每个任务结束后口头追问 1-2 个问题（不要问"好用吗"，要问"刚才你在犹豫什么？"）
  • Flow 8（危机响应）不做真实触发，改为：把 L2 截图打印出来或放大屏显示，直接访谈「如果你看到这个界面，你第一眼注意到什么？你会怎么做？」

二、产品内嵌采集（持续运营中的 UI 设计）

按照 A7 的 8 条流程，有明确的反馈触发时机。这些是需要在 UI 里增加的轻量反馈组件，不影响主流程：

① 对话结束后 — 情感反馈浮层 P0

对话自然结束或用户点返回时，小星出来问一句：「今天聊得怎么样？」配 3 个表情选项（很好 / 还行 / 有点难受）。不用文字填写，3秒可完成。这是最高价值的定性信号。

埋点对应：dialog_end_emotion_feedback

② 首次完成核心操作后 — 新手评分气泡 P0

Flow 2 第一次语音识别成功后、Flow 4 第一次切换 IP 后，小星冒出：「刚才的操作顺畅吗？」配 👍👎 两个按钮。时机精准，不打断主流程。

埋点对应：first_operation_smoothness

③ 聊愈室结束时 — 沉浸感确认 P1

点「结束对话」退出聊愈室时，插入一个轻量卡片：「这次聊愈室让你感觉…」配几个情绪词选项（平静了 / 还是很乱 / 不太适应）。这直接对应 Flow 5 的沉浸感评测指标。

埋点对应：liaoyu_immersion_feedback

④ 清空记录前 — 意图确认 + 数据收集 P1

用户点「清空聊天记录」时，确认弹窗里加一个可选原因：「为什么想清空？」（换个新开始 / 隐私保护 / 聊乱了 / 其他）。不强制填写，但能采集到 Flow 7 里跨IP混排造成的困惑信号。

埋点对应：chat_clear_reason

⑤ 定期 NPS 收集 — 7天后触发 P2

用户累计使用满 7 天，在首页小星卡片上触发一次 NPS 打分（0-10 推荐指数），配一个开放文本框。只触发一次，不重复打扰。

埋点对应：nps_7day_score
实施优先级：①② 是最小可行版（P0），做完就能拿到核心数据。③④⑤ 是有余力时再做（P1/P2）。UI 改动量都很小，本质上是在现有节点上挂轻量反馈组件，不影响主流程。
被动数据采集方案（无需线下观察员的规模化采集）

如果无法进行线下有主持人的用研测试，可以通过以下三类被动采集方式，结合 A7 已有的埋点，来推断易用性问题。被动数据能告诉你「哪里有问题」，但很难告诉你「为什么有问题」和「用户当时在想什么」。

一、行为数据分析（已有埋点的深度利用）

不需要新技术栈，直接从现有埋点里挖掘卡点信号：

① 任务完成时长异常检测

  • entry_find_time > 20秒的用户占比 → Flow 1 入口发现性差
  • ip_selection_browse_time > 60秒 → Flow 4 三个IP区分度不清晰
  • voice_button_hold_start 到 voice_button_hold_end < 0.5秒的高频重试 → Flow 2 用户误以为是"点击"而非"按住"

② 放弃率 & 回退率

  • 进入对话界面后 5秒内退出（无任何输入） → 入口预期与落地页不匹配
  • ip_switch_tap 后 1分钟内再次触发 ip_switch_tap → 切换后发现不是想要的，盲选
  • chat_clear_tap 后的 cancel 高比例 → Flow 7 确认弹窗文案不清晰

③ 错误操作模式识别

  • 连续3次 voice_button_hold_start 间隔 < 2秒 → 用户在重试，推测识别失败或不知道怎么用
  • cross_ip_confusion_signal（切换IP后1分钟内再开设置）→ Flow 7 跨IP混排造成困惑
  • session_tag_display 曝光后用户立即向上滚动查看历史 → 用户在找"新对话从哪开始"

二、会话回放 & 热力图（需轻量工具）

如果能加入类似 Hotjar/FullStory 的轻量 SDK（移动端也支持），可以看到：

会话回放（Session Replay）

  • 筛选 first_voice_attempt_success = false 的会话，回放看用户到底在哪卡住了（是没找到按钮？还是按了但没按住？）
  • 筛选 Flow 8 触发后 crisis_card_dismissed 的会话，看用户是直接划走还是犹豫后才关闭

热力图（Heatmap）

  • 首页的点击热力图 → 验证小星卡片是否在视觉焦点区
  • IP选择界面的停留热力分布 → 看用户是平均浏览三个IP，还是只看第一个就选了

无效点击追踪（Rage Clicks）

  • 短时间内连续点击同一区域但无响应 → 可能是把"按住说话"当成了"点击"，或者以为某个文字是链接但其实不是

三、文本反馈的 NLP 分析

如果「二、产品内嵌采集」的开放文本框收集到了用户评论，可以做关键词提取：

高频负面词云

  • "找不到" / "不知道怎么" → 发现性问题
  • "乱了" / "不知道在跟谁聊" → Flow 7 跨IP混排
  • "没反应" / "一直识别不了" → Flow 2 语音交互
  • "吓到了" / "突然弹出" → 换装弹窗或聊愈室入口的打断感

情感分析分值 < 3分的会话

拉取对应的埋点序列，看操作路径有没有共性（比如都在 ip_switch_tap 后情感分下降）

四、A/B 测试验证假设

当你从上述数据发现疑似问题后，可以小流量 A/B 验证：

假设：Flow 2 用户不理解"按住"，是因为按钮没有明确引导文案

实验：A组（现状），B组（按钮下方增加小字"按住说话"）

观察：B组的 first_voice_attempt_success 是否显著提升

⚠️ 局限性提醒

被动数据能告诉你"哪里有问题"，但很难告诉你"为什么有问题"和"用户当时在想什么"。

举例：
• 埋点显示 ip_selection_browse_time 平均 45秒 → 你知道用户在犹豫，但不知道是"三个IP介绍太长看不完"还是"看完了但还是不懂区别"
• 会话回放看到用户在 Flow 8 安全卡片停留 10秒后划走 → 不知道是"觉得不需要"还是"不敢点怕被记录"

所以建议混合使用：
✓ 用被动数据筛选高优先级问题（哪个流程卡点最严重？）
✓ 用线下观察深挖根因（为什么卡？具体哪句话没看懂？）
✓ 用A/B 测试验证解决方案（改了之后真的好了吗？）

对于 AI星球这种学校采购的产品，建议先跑 1-2 个月被动数据，找出 Top 3 卡点，再针对性地去 1-2 所试点学校做深度访谈。这样用研效率最高。
