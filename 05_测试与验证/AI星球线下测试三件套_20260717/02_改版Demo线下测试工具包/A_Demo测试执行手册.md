# 改版 Demo 线下测试 · 执行手册

适用阶段：7-8 月产出优化 demo 后  
测试对象：2-3 条优化主线 demo  
单人建议时长：25-35 分钟  
核心目标：验证改版 demo 是否比当前 APK 更容易理解、更愿意开始、更有被理解感，并判断是否值得进入正式迭代。

## Demo 测试和 APK 测试的区别

APK 测试是找问题，Demo 测试是验证方案。

因此 Demo 测试不需要覆盖所有功能，也不需要复刻 APK 的 7 个 Flow。它只需要围绕改版主线回答：

1. 学生是否看得懂这条新路径？
2. 学生是否愿意开始？
3. 学生是否能完成关键动作？
4. 学生是否感到被理解？
5. 学生是否愿意下次继续用？
6. Demo 是否引入了新的困惑？

## 建议测试主线

本工具包默认支持 3 条主线。实际可根据 demo 产出情况选择其中 2-3 条。

### 主线 A：首页启动路径

首页 → 情绪打卡 → AI 接住 → 今日推荐

要验证的问题：

- 学生打开后是否知道第一件事做什么。
- 情绪打卡是否成为自然起点。
- AI 伙伴接住是否让学生感觉被理解。
- 今日推荐是否像“帮我下一步”，而不是任务列表。

### 主线 B：AI 伙伴路径

AI 对话 → 三 IP 理解/选择 → 继续聊 → 情绪记忆

要验证的问题：

- 学生是否理解小星、吉塔、罗素分别适合什么。
- 学生是否愿意主动说出真实困惑。
- “它记得我”的设计是否被感知。
- 三 IP 是否像不同伙伴，而不是同一个入口换皮。

### 主线 C：完成反馈路径

脑之岛/测评/放松训练 → 完成反馈 → 成长记录 → AI 回应

要验证的问题：

- 学生完成一个动作后，是否知道自己获得了什么。
- 成果是否能回到首页、AI、成长记录中。
- 完成反馈是否增强继续使用意愿。
- 成长记录是否清楚、有价值、不像任务流水账。

## 样本建议

建议 8-12 人。

年级配额：

- 小学高年级：2-3 人
- 初中：4-6 人
- 高中：2-3 人

如果 demo 明显偏高中生场景，例如选科/生涯规划，可增加高中样本比例。

## 测试材料准备

- Demo 可稳定运行，避免现场频繁崩溃。
- 每条主线准备固定入口。
- 准备 2-3 个测试账号或假数据状态。
- 主持人提前熟悉 demo 分支。
- 每条主线的预期关键动作写在记录表中。

## 开场话术

今天给你看的不是正式上线版本，而是我们正在设计的新体验样机。  
它可能有些地方还不能点，或者内容还不完整，这都没关系。  
我们想知道的是：这个方向你看不看得懂、愿不愿意用、哪里让你困惑。

如果你觉得“不像学生会用的东西”，请直接说出来。

## 主持原则

- 不解释设计意图，先看学生是否自然理解。
- 不要求学生夸 demo。
- 遇到不能点的地方，可以说“这里现在是样机，我们先看你以为它会发生什么”。
- 优先记录真实反应，而不是完成全部步骤。
- 每条主线结束后立刻问 3 个轻量问题。

## 每条主线结束后的 3 个问题

1. 这条流程顺不顺？1-5 分。
2. 有没有哪一步你不知道为什么要做？
3. 如果这是正式 App，你愿意下次继续用这条功能吗？为什么？

## Demo 测试的结果分类

测试后不要只判断“喜欢/不喜欢”，要归因到以下类型：

| 类型 | 含义 | 后续动作 |
| --- | --- | --- |
| 明显改善 | 学生更快理解、更少提示、更愿意继续 | 进入开发候选 |
| 方向正确但表达不清 | 学生认可价值，但看不懂入口或文案 | 继续优化设计 |
| 学生无感 | 设计团队认为重要，但学生没感知 | 降低优先级或换表达 |
| 引入新困惑 | Demo 解决旧问题但带来新成本 | 重新设计 |
| 原型限制导致误判 | 因 demo 不完整导致卡住 | 需二次验证 |

## 不建议在 Demo 测试中做的事

- 不做完整 SUS。Demo 不是完整系统，SUS 容易误导。
- 不测试过多支线功能。
- 不让学生扮演危机用户。
- 不把视觉喜好当成唯一结论。
- 不用“你觉得新版是不是更好”这种诱导问法。

## 最终产出

Demo 测试结束后，产出：

- 每条主线的有效性结论
- 学生能否理解的证据
- 学生愿不愿意继续用的证据
- Demo 新增困惑点
- 进入正式迭代的建议列表
