# APK vs Demo 对比分析框架

适用阶段：完成 APK 当前版本测试和 Demo 测试之后  
目标：判断 demo 是否真正改善当前 APK 的关键问题，并决定哪些主线进入正式迭代。

## 对比分析的核心问题

不要只问“学生更喜欢哪个版本”。  
要回答：

1. Demo 是否减少了当前 APK 的核心卡点？
2. Demo 是否提升了学生的理解、开始、完成和继续使用意愿？
3. Demo 是否带来了新的理解成本？
4. Demo 的改善是否足够强，值得进入开发？

## 两轮数据如何对齐

APK 测试和 Demo 测试不一定是同一批学生，也不一定测完全相同任务。因此对比时不强行逐项一一对应，而是围绕“主线假设”对齐。

### 主线 A：首页启动路径

对应 APK 问题：

- P1 首屏双大卡片权重相当，无视觉主角
- P2 首页 AI 双入口认知存疑
- P11 任务缺乏情境化推送

Demo 目标：

- 第一行动更清楚
- 情绪打卡成为自然起点
- AI 接住和今日推荐形成下一步

### 主线 B：AI 伙伴路径

对应 APK 问题：

- P2 AI 无全局持续入口
- P3 对话无情感记忆
- P9 免责声明削弱信任感

Demo 目标：

- 三 IP 更容易理解
- 对话更像伙伴而非工具
- 记忆设计带来信任，而不是冒犯

### 主线 C：完成反馈路径

对应 APK 问题：

- P5 脑之岛成就不外溢
- P6 解忧信箱缺乏关闭反馈
- P8 星星商店激励闭环不清

Demo 目标：

- 完成后知道自己获得了什么
- 成果能回到首页、AI、成长记录
- 增强下次使用意愿

## 统一指标

建议所有主线都按以下指标对比。

| 指标 | APK 记录 | Demo 记录 | 判断方式 |
| --- | --- | --- | --- |
| 任务完成率 | 成功 / 需提示 / 失败 | 成功 / 需提示 / 失败 | Demo 成功率是否提升 |
| 需提示率 | 需提示人数 / 总人数 | 需提示人数 / 总人数 | Demo 是否减少主持提示 |
| 首次停顿点 | 触点 + 原话 | 触点 + 原话 | 旧卡点是否消失 |
| 关键路径耗时 | 秒/分钟 | 秒/分钟 | 是否更快完成，不牺牲理解 |
| 即时满意度 | 1-5 分 | 1-5 分 | Demo 是否提升 |
| 被理解感 | 1-5 分 | 1-5 分 | AI/情绪路径核心指标 |
| 继续使用意愿 | 1-5 分 | 1-5 分 | 是否有留存潜力 |
| 新增困惑 | 无/有 | 无/有 | Demo 是否引入新问题 |

## 改善判定标准

### 强改善

满足至少 3 条：

- 完成率提升明显
- 需提示率下降明显
- 关键卡点原话减少
- 满意度提升 1 分以上
- 被理解感提升 1 分以上
- 继续使用意愿提升 1 分以上
- 没有明显新增困惑

建议：进入正式迭代候选。

### 中等改善

满足 1-2 条，但仍有明显困惑。

建议：方向保留，继续打磨入口、文案或信息层级。

### 弱改善或无改善

学生看不出差异，或只是觉得“更好看”但不会更愿意用。

建议：暂缓进入开发，重新定义问题。

### 负向改善

Demo 解决一个旧问题，但引入更大的新问题。

建议：不进入开发，回到设计阶段。

## 典型结论写法

### 好结论示例

主线 A 首页启动路径在 Demo 中表现为强改善。APK 中 8/10 名学生在首屏无法判断第一行动，Demo 中仅 2/10 名学生出现停顿；即时满意度从 3.1 提升到 4.2。学生原话从“这里好多东西，不知道先点哪个”转变为“它好像先问我今天怎么样”。建议进入下一阶段开发。

### 不充分结论示例

“学生更喜欢新版首页。”  
这个结论不够，需要补充喜欢的原因、是否更容易开始、是否减少停顿、是否愿意继续用。

## 决策矩阵

| 主线 | 改善程度 | 新增困惑 | 开发成本 | 风险 | 建议 |
| --- | --- | --- | --- | --- | --- |
| 首页启动路径 | 强 / 中 / 弱 / 负向 | 高 / 中 / 低 | 高 / 中 / 低 | 高 / 中 / 低 | 进入开发 / 继续打磨 / 暂缓 |
| AI 伙伴路径 | 强 / 中 / 弱 / 负向 | 高 / 中 / 低 | 高 / 中 / 低 | 高 / 中 / 低 | 进入开发 / 继续打磨 / 暂缓 |
| 完成反馈路径 | 强 / 中 / 弱 / 负向 | 高 / 中 / 低 | 高 / 中 / 低 | 高 / 中 / 低 | 进入开发 / 继续打磨 / 暂缓 |

## 注意事项

- 两轮测试样本不完全一致时，不要过度解释小数差异。
- 小学、初中、高中要分层看，不要只看总平均。
- Demo 原型不完整导致的问题要单独标注，避免误判设计方向。
- 对 AI 记忆、隐私、危机响应这类敏感体验，用户“觉得不舒服”的原话优先级高于分数。
- 视觉偏好不能单独作为开发依据，必须和可用性/理解度指标一起看。
